**【0:00 - 0:30】开场与业务背景**

**【0:30 - 1:30】架构规范与配置演示（拿工程规范分）**

> "首先展示代码规范与工程结构。
>
> 项目采用标准的 **Controller → Service → DAO → Model** 四层分层架构，所有依赖通过 `main.go` 统一注入（Dependency Injection），每一层都定义了接口（Interface），方便后续替换实现和单元测试 Mock。
>
> 【切换 IDE，打开 `backend/` 目录树】
>
> 中间件链方面，我按顺序注册了 6 个中间件：
> 1. **Recovery 中间件** — 使用 `safego` 包封装的安全 goroutine 启动器，捕获任何 panic 并记录完整堆栈，不会导致整个服务崩溃；
> 2. **RequestID 中间件** — 基于 `github.com/google/uuid`，为每个请求注入唯一追踪 ID，贯穿整个调用链；
> 3. **CORS 中间件** — 可配置的跨域策略，允许前端独立部署时正常访问；
> 4. **Logger 中间件** — 接入 **Zap**（`go.uber.org/zap`），以 JSON 格式结构化记录每个请求的方法、路径、状态码和响应耗时。
>
> 【打开 `backend/pkg/zap/zap.go`】
>
> 日志系统基于 **Zap** + **Lumberjack**（`gopkg.in/natefinch/lumberjack.v2`）实现：
> - 采用 `zapcore.NewTee` 同时输出到控制台和文件；
> - JSON 编码器确保日志可被 ELK 等日志平台直接采集；
> - Lumberjack 按天自动切割，配置了单文件最大 100MB、保留 30 个备份、最长保留 7 天的滚动策略，旧日志自动 gzip 压缩。
>
> 【打开 `backend/pkg/config/config.go` 和 `backend/conf/config.yaml`】
>
> 配置管理使用 **Viper**（`github.com/spf13/viper`）：
> - 所有环境参数（数据库连接、Redis 地址、JWT 密钥等）集中在 `config.yaml` 中管理；
> - 支持环境变量覆盖，前缀为 `LAB_`，并自动将点号转为下划线（如 `LAB_JWT_SECRET` 覆盖 `jwt.secret`），实现配置与代码的完全分离，容器化部署时无需修改代码。"

**【1:30 - 2:30】认证、权限与参数校验（拿功能满分）**

> "接下来演示核心安全机制。
>
> 【切换到 Postman，发送 POST `/api/v1/auth/login`】
>
> 首先演示登录认证。用户提交账号密码后，系统使用 **golang.org/x/crypto/bcrypt** 对密码进行哈希比对——数据库中存储的是 bcrypt 加密后的密文，从不保存明文密码。验证通过后，使用 **golang-jwt/jwt/v5** 生成 JWT Token：
> - 签名算法采用 **HMAC-SHA256**；
> - Claims 中嵌入了 `user_id`、`username`、`role_id`、`role_name`，后续鉴权无需再查数据库；
> - Token 有过期时间限制，由配置文件中的 `jwt.expire` 控制。
>
> 【打开 `backend/router/middleware/auth.go`】
>
> Auth 中间件从请求头提取 `Bearer Token`，调用 `jwt.ParseWithClaims` 解析并校验签名和过期时间，校验通过后将用户信息注入 Gin Context。
>
> 【打开 `backend/router/middleware/ratelimit.go`】
>
> 此外，登录接口还单独挂载了 **Redis 限流中间件**（`github.com/redis/go-redis/v9`），采用固定窗口算法，限制每个 IP 每分钟最多 10 次登录请求，有效防止暴力破解。"
>
> ---
>
> "接下来说明参数校验。
>
> 【打开 `backend/service/auth.go`，展示 struct tag】
>
> 所有请求体结构体都使用了 **go-playground/validator** 的 binding 标签（如 `binding:"required,min=3,max=32"`），由 Gin 框架在 Controller 层自动完成绑定与校验。参数不合法时直接在控制层返回 400，不会进入业务逻辑层。
>
> 同时，项目定义了统一的错误码体系（`backend/pkg/errcode/errcode.go`），覆盖 1xxx（用户）、2xxx（设备）、3xxx（借用）、4xxx（权限）、5xxx（系统），配合统一响应格式（`backend/pkg/response/response.go`），所有接口返回结构一致，前端处理非常方便。"
>
> ---
>
> "现在演示权限控制——这是本系统的核心亮点之一。
>
> 【打开 `backend/cmd/main.go`，定位 `initCasbin()` 函数】
>
> RBAC 鉴权使用 **Casbin v2**（`github.com/casbin/casbin/v2`）+ **GORM Adapter v3**（`github.com/casbin/gorm-adapter/v3`）：
> - Casbin 的模型定义在 `conf/rbac_model.conf` 中，采用 `g(r, p)` 的 RBAC 标准模型；
> - 策略（Policy）存储在 MySQL 的 `casbin_rule` 表中，由 GORM Adapter 自动同步，策略变更即时生效无需重启；
> - 系统预置了 5 个角色（超级管理员 → 实验室负责人 → 设备管理员 → 普通成员 → 访客），通过 `g` 实现角色继承链，上级自动拥有下级的全部权限。
>
> 【打开 `backend/router/middleware/casbin.go`】
>
> Casbin 中间件挂载在所有 `/api/v1/*` 受保护路由上，从 Gin Context 中提取用户角色名，调用 `enforcer.Enforce(sub, obj, act)` 进行鉴权：
> - 请求路径按正则匹配策略（如 `super_admin` 可访问 `/api/v1/*` 的所有方法）；
> - 被拒绝的请求会以结构化日志（Zap）记录 `user_id`、`path`、`method`，便于审计追溯。
>
> 【切换到 Postman，先用普通成员 Token 调 POST `/api/v1/borrow/approve`】
>
> "现在我用'普通成员'的 Token 尝试调用'负责人审核'接口——系统返回 **403 Forbidden**，错误码 `4001`（权限不足），Casbin 鉴权拦截生效。
>
> 再用'实验室负责人'的 Token 调用同一接口——审批成功，返回 200。这就是 RBAC 的实际落地效果。"

**【2:30 - 3:30】核心闭环：事务与缓存优化（拿高阶优化分）**

> "最后演示设备借阅的完整业务闭环——这是系统中最复杂、也是最能体现数据库功底的模块。
>
> 【打开 `backend/service/equipment.go`，展示缓存逻辑】
>
> **第一层优化：Redis 缓存（Cache-Aside 模式）**
>
> 首页的'可用设备列表'是最高频请求。系统使用 **go-redis v9** 实现了 Cache-Aside 策略：
> - 请求到达时，先从 Redis 读取缓存（`GET equipment:list:...`），命中则直接返回，数据库零压力；
> - 缓存未命中时查询 MySQL，结果写入 Redis 并设置 TTL（过期时间），下次请求即可命中；
> - 当设备被借出、归还或信息修改时，主动 **删除（Invalidate）** 对应缓存，保证数据的最终一致性。
>
> Redis 连接池配置了最大连接数、最小空闲连接、超时重试等参数，确保高并发下的稳定性。
>
> 【打开 `backend/service/borrow.go`，展示事务逻辑】
>
> **第二层优化：GORM 事务 + 行级锁**
>
> 借阅审批是典型的多表写操作——需要同时完成两件事：
> 1. 更新借阅记录状态为"已通过"；
> 2. 扣减设备的可用库存（`available--`）。
>
> 这两个操作必须原子执行：任何一个失败，另一个必须回滚，否则就会出现"审批通过了但库存没扣"或"库存扣了但审批状态没变"的数据不一致。
>
> 系统在 **GORM** 中通过 `db.Transaction(func(tx *gorm.DB) error { ... })` 开启数据库事务：
> - 事务内使用 `clause.Locking{Strength: "UPDATE"}`（即 `SELECT ... FOR UPDATE`）对借阅记录和设备行加行级排他锁，防止并发审批时出现超借；
> - 两条写操作共享同一个 `*gorm.DB` 事务实例（通过 DAO 层的 `*WithTx` 方法），确保同时成功或同时失败。
>
> 【打开 `backend/dao/equipment.go`，展示 FOR UPDATE 实现】
>
> DAO 层封装了带锁查询方法，使用 GORM 的 `clause.Locking` 子句生成 `SELECT ... FOR UPDATE` SQL，在高并发场景下保证库存扣减的准确性。"
>
> ---
>
> "最后补充数据库层面的优化：
>
> 【打开 `backend/cmd/main.go`，展示连接池配置】
>
> - MySQL 连接池：通过 `SetMaxIdleConns(10)`、`SetMaxOpenConns(100)`、`SetConnMaxLifetime(time.Hour)` 配置了空闲连接数、最大连接数和连接最大存活时间；
> - 高频查询字段（如 `equipment.status`、`borrow_record.user_id`、`borrow_record.status`）均在 Model 层建立了数据库索引；
> - GORM 的日志模式设为 `Warn` 级别，避免生产环境日志刷屏，同时配合 Zap 记录所有慢查询。
>
> 【切换到 Postman，完整走一遍借阅流程：申请 → 审批 → 查看状态变化】
>
> 可以看到整个流程数据一致、缓存即时失效、权限各司其职。"

> 📌 **技术点速查：**
>
> | 用到的库 | 用在什么地方 |
> |---|---|
> | `gorm.io/gorm` + `gorm.io/driver/mysql` | ORM、事务、FOR UPDATE 行锁、连接池、AutoMigrate |
> | `github.com/redis/go-redis/v9` | Cache-Aside 缓存策略、缓存失效、连接池 |
