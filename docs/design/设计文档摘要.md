
### 二、 详细设计文档 (架构与技术实现)

1. 技术栈选型映射 本项目严格遵循课程要求，不做任何核心框架替换，具体映射如下：

* **Web 框架：** `Gin` (负责路由分发与请求上下文处理)
* 
**配置管理：** `Viper` (解析 `config.yaml`，实现配置与代码分离) 


* **身份认证：** `JWT` (生成与解析 Token，配置过期时间)
* **权限控制：** `Casbin` (结合 GORM 适配器，实现 RBAC 权限模型拦截)
* **参数校验：** `Validator` (使用结构体 Tag 校验前端传入的 JSON 数据)
* 
**日志规范：** `Zap` (实现结构化日志、按天进行日志分割，区分 Info/Error 级别) 


* 
**数据库层：** `GORM` + `MySQL` (实现连接池化、多表关联、事务处理) 


* 
**缓存层：** `Redis` (设备列表的高频查询缓存、JWT Token 黑名单注销机制) 



2. 分层架构设计 (高内聚低耦合) 

* `cmd/`：项目入口 `main.go`
* `conf/`：配置文件 `config.yaml`
* `router/`：路由注册与中间件挂载（Auth, Casbin, Recovery）
* `controller/`：解析请求参数（Validator），调用 Service，统一 JSON 响应格式
* `service/`：核心业务逻辑处理（如借阅工单的流转）
* `dao/`：基于 GORM 的数据库操作，封装事务
* 
`models/`：GORM 实体模型，定义表结构与索引优化


* 
`pkg/`：公共组件（Zap 初始化、Redis 连接池、全局异常处理）



3. 数据库核心表结构 (ER 关联) 

* `sys_users`: id, username, password_hash, role_id
* `sys_roles`: id, role_name (对应 Casbin 策略)
* 
`lab_equipments`: id, name, total_stock, available_stock (需建立适当的索引优化查询)


* `borrow_records`: id, user_id, equipment_id, status(申请中/已借出/已归还)

4. 工程化规范实践 

* **全局异常处理：** 在 Gin 中间件编写 `Recovery`，捕获 Panic 并通过 Zap 记录堆栈信息，给前端返回标准的 `{"code": 500, "msg": "系统内部错误"}`。
* 
**数据库优化：** 在初始化 GORM 时配置 `SetMaxIdleConns` 和 `SetMaxOpenConns` 实现连接池化。
