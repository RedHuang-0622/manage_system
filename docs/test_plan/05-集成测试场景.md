# 05 — 集成测试场景

## 1 测试环境

集成测试需要真实的外部依赖：

```
docker-compose up -d              # 启动 MySQL + Redis
go test ./... -tags=integration   # 运行集成测试
```

**配置：** `conf/config.test.yaml`（独立测试库 `lab_manage_test`）

## 2 数据准备

每个集成测试场景使用 `SetupTest` + `TeardownTest` 确保隔离：

```go
func SetupIntegrationTest(t *testing.T) (*gorm.DB, *redis.Client) {
    db := connectTestDB()
    db.AutoMigrate(&models.SysUser{}, &models.SysRole{},
        &models.LabEquipment{}, &models.BorrowRecord{})

    // 种子数据
    db.Create(&models.SysRole{ID: 1, RoleName: "super_admin"})
    db.Create(&models.SysRole{ID: 2, RoleName: "lab_admin"})
    db.Create(&models.SysRole{ID: 3, RoleName: "member"})

    rdb := redis.NewClient(&redis.Options{Addr: "localhost:6380"})
    rdb.FlushDB(ctx)

    return db, rdb
}
```

## 3 集成测试场景

### 场景 1：完整借阅闭环（Happy Path）⭐⭐⭐

**测试流程：**

```
Step 1: 创建用户
  POST /api/v1/users {username:"member1", password:"123456", role_id:3}
  → 201, user_id=3

Step 2: 用户登录
  POST /api/v1/auth/login {username:"member1", password:"123456"}
  → 200, token=<member_token>

Step 3: 管理员登录
  POST /api/v1/auth/login {username:"admin", password:"admin123"}
  → 200, token=<admin_token>

Step 4: 创建设备
  POST /api/v1/equipments {name:"GPU Server", total_stock: 3}
  → 201, equip_id=1, available_stock=3

Step 5: 验证设备大厅（读缓存）
  GET /api/v1/equipments?page=1&page_size=10
  → 200, total=1, available_stock=3

Step 6: 发起借阅申请
  POST /api/v1/borrows/apply {equipment_id:1, quantity:2}
  → 201, borrow_id=100, status="申请中"

Step 7: 验证库存未变化（申请不扣库存）
  GET /api/v1/equipments/1
  → available_stock = 3

Step 8: 管理员审批通过
  POST /api/v1/borrows/100/approve {approve:true}
  → 200, status="已借出"

Step 9: 验证库存已扣减（事务提交）
  GET /api/v1/equipments/1
  → available_stock = 1 (= 3 - 2)

Step 10: 验证缓存已失效（第二次 GET 不走缓存）
  GET /api/v1/equipments/1
  → available_stock = 1 (来自 DB)

Step 11: 归还设备
  POST /api/v1/borrows/100/return
  → 200, status="已归还"

Step 12: 验证库存已恢复
  GET /api/v1/equipments/1
  → available_stock = 3

Step 13: 验证我的借阅记录
  GET /api/v1/borrows/my
  → 200, total=1, list[0].status="已归还"
```

### 场景 2：审批拒绝流程

```
Step 1: 发起申请 → borrow_id=101, status="申请中"
Step 2: 审批拒绝
  POST /api/v1/borrows/101/approve {approve:false, approve_note:"设备已预约"}
  → 200, status="被拒绝"
Step 3: 验证库存未变化
  GET /api/v1/equipments/1
  → available_stock 不变
Step 4: 再次审批（终态校验）
  POST /api/v1/borrows/101/approve {approve:true}
  → 400 {code:3007} "工单状态异常"
```

### 场景 3：并发审批防超卖 ⭐⭐⭐

```
准备：设备 available_stock=3，创建 5 个"申请中"工单
操作：5 个 goroutine 并发 POST /borrows/{id}/approve {approve:true}

验证：
  ├── 成功数 = 3
  ├── 失败数 = 2，全部返回 {code:3006} "库存不足"
  ├── 最终 available_stock = 0（不超卖，不为负）
  └── borrow_records 中"已借出" = 3，"申请中" = 2（审批失败的保持原状态）
```

### 场景 4：权限越界拦截

```
Step 1: member 登录获取 Token
Step 2: member 尝试管理用户
  POST /api/v1/users {username:"test", ...} (member Token)
  → 403 {code:2005}
Step 3: member 尝试审批
  POST /api/v1/borrows/1/approve {approve:true} (member Token)
  → 403 {code:2005}
Step 4: member 尝试下架设备
  DELETE /api/v1/equipments/1 (member Token)
  → 403 {code:2005}
Step 5: member 正常操作不受影响
  GET /api/v1/equipments (member Token)
  → 200
  POST /api/v1/borrows/apply {equipment_id:1, quantity:1} (member Token)
  → 201
```

### 场景 5：JWT 黑名单机制

```
Step 1: 用户登录 → token_A
Step 2: 使用 token_A 正常请求
  GET /api/v1/equipments
  → 200
Step 3: 用户登出
  POST /api/v1/auth/logout (token_A)
  → 200 {code:0}
Step 4: 使用已登出 Token
  GET /api/v1/equipments (token_A)
  → 401 {code:2004} "Token已失效"
Step 5: 刷新旧 Token 后旧 Token 失效
  POST /api/v1/auth/refresh {token: token_A}
  → 200, 新 token_B
Step 6: 旧 token_A 再次请求
  GET /api/v1/equipments (token_A)
  → 401 {code:2004}
Step 7: 新 token_B 正常请求
  GET /api/v1/equipments (token_B)
  → 200
```

### 场景 6：缓存一致性验证

```
Step 1: 首次查询设备列表（写缓存）
  GET /api/v1/equipments
  → Redis 中存在 equip:list:1:12:... key

Step 2: 创建新设备（触发缓存失效）
  POST /api/v1/equipments {name:"新设备", total_stock:5}

Step 3: 再次查询列表（走 DB 重建缓存）
  GET /api/v1/equipments
  → 包含新设备

Step 4: 更新设备详情（触发详情缓存失效）
  PUT /api/v1/equipments/1 {location:"新位置"}

Step 5: 查询设备详情（走 DB）
  GET /api/v1/equipments/1
  → location="新位置"

Step 6: 等待 TTL 过期（>30s）
  GET /api/v1/equipments
  → 从 DB 重建缓存，数据正确
```

### 场景 7：软删除与关联保护

```
Step 1: 有未归还借阅的设备 → 尝试下架
  DELETE /api/v1/equipments/1
  → 400 错误（有未归还借阅）

Step 2: 归还后下架
  POST /api/v1/borrows/100/return
  DELETE /api/v1/equipments/1
  → 200, status=0

Step 3: 下架后不可借阅
  POST /api/v1/borrows/apply {equipment_id:1, quantity:1}
  → 400 错误（设备已下架）
```

### 场景 8：重复借阅拦截

```
Step 1: member1 申请设备 1 → 201
Step 2: member1 再次申请同一设备 1 → 400 {code:3005} "已有该设备的未归还借阅"
Step 3: 审批拒绝第一个申请
Step 4: member1 可以重新申请同一设备 → 201（旧记录已终态）
```

### 场景 9：库存联动更新

```
Step 1: 创建设备 total=10, available=10
Step 2: 审批借出 3 → available=7
Step 3: 管理员更新 total_stock 从 10 改为 8
  → new_available = 7 + (8-10) = 5 ✓
Step 4: 管理员尝试 total_stock 从 8 改为 3
  → new_available = 5 + (3-8) = 0 ✓（恰好合法）
Step 5: 管理员尝试 total_stock 从 3 改为 2
  → new_available = 0 + (2-3) = -1 ✗ → 错误 "库存不足"
```

### 场景 10：禁止禁用自身

```
Step 1: 管理员登录
Step 2: 管理员尝试禁用自己
  DELETE /api/v1/users/1 (自己的 ID)
  → 400 错误
Step 3: 管理员禁用其他用户
  DELETE /api/v1/users/3
  → 200
```

## 4 集成测试运行清单

```bash
# 启动环境
docker-compose -f docker-compose.test.yaml up -d

# 等待 MySQL 就绪
until docker exec mysql_test mysqladmin ping -h localhost; do sleep 1; done

# 运行全部集成测试
go test ./... -tags=integration -v -count=1

# 单独运行某个场景
go test ./tests/ -tags=integration -v -run TestScenario1_FullBorrowFlow

# 并发测试（更长时间）
go test ./tests/ -tags=integration -v -run TestScenario3_Concurrent -timeout 60s

# 清理
docker-compose -f docker-compose.test.yaml down -v
```
