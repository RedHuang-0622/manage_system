# 01 — 身份与访问管理（IAM）

## 1 模块定位

身份与访问管理（Identity & Access Management）是系统的安全基石，负责"你是谁"（认证）和"你能做什么"（鉴权）两大核心职责。本模块将用户生命周期管理、JWT 认证、Casbin RBAC 权限控制合并为一个有界上下文，因为它们共享 `sys_users` + `sys_roles` + `casbin_rule` 三张表，且 Auth 中间件与 Casbin 中间件在请求链上紧耦合。

**涉及的包/文件：**
- `controller/auth.go` + `controller/user.go` — HTTP 层
- `service/auth.go` + `service/user.go` — 业务逻辑
- `dao/user.go` + `dao/role.go` — 数据访问层
- `models/user.go` — SysUser / SysRole 实体
- `router/middleware/auth.go` — JWT 认证中间件
- `router/middleware/casbin.go` — Casbin 鉴权中间件
- `pkg/jwt/jwt.go` — JWT 工具
- `conf/rbac_model.conf` — Casbin 模型定义

### 2.5 预置角色

| ID | role_name | description | 说明 |
|----|-----------|-------------|------|
| 1 | super_admin | 超级管理员（指导老师） | 全局权限 |
| 2 | lab_admin | 实验室负责人 | 管理成员、审批借阅、管理设备 |
| 3 | member | 普通成员 | 浏览设备、发起借阅、查看自己记录 |

### 2.6 role_name 全链路依赖（关键约束）

`role_name` 在系统中有四个出现位置，必须保持一致，任一环节断裂都会导致鉴权失败：

```
sys_roles.role_name          -- DB 存储（如 "member"）
    │
    ▼
JWT Claims.role_name         -- 登录时从 DB 读取后写入 Token
    │  (pkg/jwt/jwt.go → GenerateToken)
    ▼
Auth 中间件 → c.Set("role_name", claims.RoleName)
    │  (router/middleware/auth.go)
    ▼
Casbin 中间件 → Enforce(sub=role_name, obj=path, act=method)
    │  (router/middleware/casbin.go)
    ▼
casbin_rule.v0 (策略主体)    -- 必须与 role_name 字符串完全匹配
```

**约束规则：**
1. `sys_roles.role_name` 是唯一事实来源
2. 预置角色通过 `is_system = 1` 标记，DAO 层拦截对 `is_system=1` 记录的 UPDATE（禁止修改 `role_name`）和 DELETE，否则已有策略全部失效
3. 如果通过管理接口新增角色，必须同步调用 `enforcer.AddPolicy()` 写入对应策略
4. JWT Claims 中的 `role_name` 仅在登录时写入一次，之后即使角色被修改也不影响已签发 Token（设计取舍：性能 vs 实时性）

**DAO 层硬约束：**
```go
func (d *RoleDAO) Update(role *models.SysRole) error {
    if role.IsSystem == 1 && role.RoleName != old.RoleName {
        return errors.New("不允许修改系统预置角色的 role_name")
    }
    // ...
}
func (d *RoleDAO) Delete(id uint) error {
    role, _ := d.FindByID(id)
    if role != nil && role.IsSystem == 1 {
        return errors.New("不允许删除系统预置角色")
    }
    // ...
}
```

## 3 认证子模块

### 3.1 接口定义

#### 登录

```
POST /api/v1/auth/login
```

**请求：**
```json
{"username": "zhangsan", "password": "123456"}
```

**成功响应 (200)：**
```json
{"code": 0, "msg": "success", "data": {"token": "eyJ...", "expires_in": 7200}}
```

**失败响应：**
```json
{"code": 2001, "msg": "用户名或密码错误", "data": null}
{"code": 2002, "msg": "账号已被禁用", "data": null}
```

**Service 层处理流程：**
1. 通过 DAO 按 `username` 查询用户
2. 用户不存在 → ErrAuthFailed
3. `status == 0` → ErrAccountDisabled
4. `bcrypt.CompareHashAndPassword` 验证密码
5. 调用 `jwt.GenerateToken` 签发 Token（包含 user_id, username, role_id, role_name）
6. 返回 Token 和过期时间

#### 登出

```
POST /api/v1/auth/logout
Authorization: Bearer <token>
```

**成功响应：** `{"code": 0, "msg": "登出成功", "data": null}`

**实现：** 将 Token 的 SHA256 哈希写入 Redis 黑名单，Key = `jwt:blacklist:<sha256(token)>`，TTL = Token 剩余有效期（由 Redis 自动过期清理）

#### 刷新 Token

```
POST /api/v1/auth/refresh
Authorization: Bearer <old_token>
```

**请求 Body：** 无（或空 JSON `{}`）

> **v1.0 原设计缺陷：** Body 中的 `{"token": "..."}` 与 Header `Authorization: Bearer <token>` 存在歧义——两个位置的 Token 不一致时以哪个为准未定义。v1.1 起统一只从 Header 读取。

**实现要点：**
1. 从 `Authorization` Header 提取旧 Token
2. 验证旧 Token 有效（未过期、不在黑名单）
3. 从 Claims 提取用户信息，签发新 Token
4. 旧 Token 加入黑名单（SHA256 哈希）
5. 返回新 Token

### 3.2 JWT 设计

```go
type Claims struct {
    UserID   uint   `json:"user_id"`
    Username string `json:"username"`
    RoleID   uint   `json:"role_id"`
    RoleName string `json:"role_name"`
    jwt.RegisteredClaims
}
```

**配置项：**

| 配置项 | config.yaml 路径 | 默认值 | 说明 |
|--------|-----------------|--------|------|
| 密钥 | `jwt.secret` | 必填 | ≥ 32 字符 |
| 过期时间 | `jwt.expire` | 7200s | 2 小时 |
| 签发者 | `jwt.issuer` | `lab-system` | |

**pkg/jwt 包接口：**
```go
type Service struct { secret string; expire time.Duration; issuer string; redisClient *redis.Client }
func (s *Service) GenerateToken(userID uint, username string, roleID uint, roleName string) (string, int64, error)
func (s *Service) ParseToken(tokenString string) (*Claims, error)
func (s *Service) AddToBlacklist(tokenString string, expireAt time.Time) error
func (s *Service) IsInBlacklist(tokenString string) bool
```

### 3.3 Auth 中间件

**位置：** `router/middleware/auth.go`

**处理流程：**
```
提取 Authorization: Bearer <token>
  → 缺失/格式错 → 401 {code:2003, msg:"未提供认证Token"}
  → 在 Redis 黑名单 → 401 {code:2004, msg:"Token已失效"}
  → ParseToken 失败/过期 → 401 {code:2004, msg:"Token无效或已过期"}
  → 注入 Context: c.Set("user_id"), c.Set("role_name"), c.Set("username"), c.Set("role_id")
  → c.Next()
```

**白名单（跳过认证的路径）：** `POST /api/v1/auth/login`、`GET /api/v1/health`

#### 4.1.1 角色列表

**权限：** 所有登录用户

**成功响应 (200)：**
```json
{
    "code": 0, "msg": "success",

        {"id": 1, "role_name": "super_admin", "description": "超级管理员（指导老师）"},
        {"id": 2, "role_name": "lab_admin",    "description": "实验室负责人"},
        {"id": 3, "role_name": "member",        "description": "普通成员"}
    ]
}
```

#### 4.1.2 创建用户

```
POST /api/v1/users
```

**权限：** super_admin, lab_admin

**请求 Body：**
```json
{
    "username": "zhangsan",  "password": "123456",
    "real_name": "张三",     "email": "zhangsan@lab.edu.cn",
    "phone": "13800138000", "role_id": 3
}
```

**Validator 规则：**

| 字段 | 规则 |
|------|------|
| username | `required,min=3,max=32,alphanum` |
| password | `required,min=6,max=64` |
| real_name | `required,min=2,max=32` |
| email | `omitempty,email,max=64` |
| phone | `omitempty,numeric,max=16` |
| role_id | `required,gt=0` |

**成功响应 (201)：**
```json
{"code": 0, "msg": "创建成功", "data": {"id": 1, "username": "zhangsan", ...}}
```

**实现要点：**
- 密码通过 `bcrypt.GenerateFromPassword(password, 12)` 加密
- `password_hash` 标记 `json:"-"` 永不序列化
- 创建时 `status` 默认 1（启用）
- **`role_id` 必须校验存在性：** 通过 `RoleDAO.FindByID(role_id)` 确认角色存在后再创建用户，防止写入无效 role_id

#### 4.1.3 用户列表（分页）

```
GET /api/v1/users?page=1&page_size=10&keyword=&status=-1&role_id=0
```

**权限：** super_admin, lab_admin

**Query 参数：**

| 参数 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| page | int | 1 | 页码 |
| page_size | int | 10 | 每页条数（最大 100） |
| keyword | string | "" | 模糊搜索（username / real_name） |
| status | int | -1 | 1=启用, 0=禁用, -1=全部 |
| role_id | int | 0 | 角色筛选：0=全部 |

**成功响应 (200)：**
```json
{
    "code": 0, "msg": "success",
    "data": {
        "total": 50, "page": 1, "page_size": 10,
        "list": [{"id": 1, "username": "zhangsan", "role": {...}, "status": 1, ...}]
    }
}
```

**DAO 层分页查询：**
```go
func (d *UserDAO) FindPage(offset, limit int, keyword string, status int, roleID uint) ([]models.SysUser, int64, error) {
    query := d.db.Model(&models.SysUser{}).Preload("Role")
    if keyword != "" {
        like := "%" + keyword + "%"
        query = query.Where("username LIKE ? OR real_name LIKE ?", like, like)
    }
    if status >= 0 { query = query.Where("status = ?", status) }
    if roleID > 0 { query = query.Where("role_id = ?", roleID) }
    var total int64
    query.Count(&total)
    var users []models.SysUser
    err := query.Offset(offset).Limit(limit).Order("id DESC").Find(&users).Error
    return users, total, err
}
```

#### 4.1.4 用户详情

```
GET /api/v1/users/{id}
```

**权限：** super_admin, lab_admin（或本人查看自身）

**双层校验说明：**

| 层级 | 校验内容 | 实现位置 |
|------|---------|---------|
| Casbin 中间件 | 仅放行 `super_admin`、`lab_admin` 角色 | `router/middleware/casbin.go` |
| Service 层 | 若操作者非 `super_admin`/`lab_admin`，检查 `operatorID == targetUserID` | `service/auth.go` → `GetByID()` |

> **为什么不在 Casbin 做？** Casbin 基于 `(role, URL, method)` 三元组匹配，无法区分 URL 中的 `:id` 是否等于当前用户。细粒度的"本人数据"校验必须在 Service 层通过比对 `user_id` 完成。

#### 4.1.5 更新用户

```
PUT /api/v1/users/{id}
```

**权限：** super_admin, lab_admin

**请求 Body：**（全部可选，仅更新传入字段）
```json
{"real_name": "张三丰", "email": "new@lab.edu.cn", "phone": "13900139000", "role_id": 3, "status": 1}
```

**约束：** 不允许修改 `password`（另有独立接口）、不允许修改 `username`
**role_id 校验：** 若传入 `role_id`，必须通过 `RoleDAO.FindByID(role_id)` 确认角色存在

#### 4.1.6 禁用用户（软删除）

```
POST /api/v1/users/{id}/disable
```

**权限：** super_admin

**实现：** 软删除 → `status = 0`

**REST 语义说明：** 禁用用户不是删除资源（记录仍保留在数据库中），因此不使用 `DELETE` 动词。使用 `POST` + 动词化路径 `/disable` 表达"执行禁用动作"，语义更明确。

**保护规则：**
- 禁止禁用自己
- 若用户有未归还借阅，不允许禁用

#### 4.1.7 修改密码

```
PUT /api/v1/users/{id}/password
```

**权限：** super_admin 可改任意用户；其他用户仅可改本人

**双层校验说明：**

| 层级 | 校验内容 |
|------|---------|
| Casbin 中间件 | 放行所有登录用户（`member` → `PUT /api/v1/users/\d+/password`） |
| Service 层 | 若 `operatorRole != "super_admin"`，强制检查 `operatorID == targetUserID`，不匹配则返回 `2005 ErrPermissionDenied` |

> **为什么 Casbin 层放得较宽？** Casbin 无法感知 URL 路径参数 `:id` 是否等于操作者本人。安全边界由 Service 层兜底——Service 层是唯一知道 `operatorID` 和 `targetUserID` 的地方。Controller 层在调用 Service 前应从 Context 注入 `operatorID`（由 Auth 中间件写入），**不可信任请求 Body 中的任何身份字段**。

**请求 Body：**
```json
{"old_password": "123456", "new_password": "654321"}
```

**字段规则：**

| 字段 | 必填 | 说明 |
|------|------|------|
| new_password | ✅ 必填 | `min=6,max=64` |
| old_password | 条件必填 | **非 super_admin 必填**，用于验证身份；**super_admin 可省略**，省略时直接覆盖目标用户密码 |

**流程：** 非 super_admin 需验证旧密码 → bcrypt 加密新密码 → 更新 `password_hash`

**Service 层实现约束：**

> ⚠️ **安全关键：** `isAdmin` 参数必须由 Controller 从 Context 中的 `role_name` 判断（由 Auth 中间件注入），**绝不可从请求 Body 读取**。判断逻辑：`isAdmin := roleName == "super_admin"`

```go
func (s *IAMService) ChangePassword(ctx context.Context, operatorID uint, targetID uint,
    oldPwd, newPwd string, isAdmin bool) error {

    // 非管理员只能改自己
    if !isAdmin && operatorID != targetID {
        return ErrPermissionDenied
    }
    // 非管理员：必须校验旧密码
    if !isAdmin {
        if oldPwd == "" {
            return ErrInvalidParam  // old_password 不能为空
        }
        // bcrypt 验证旧密码...
    }
    // 管理员：直接更新，忽略 old_password
    // bcrypt 加密新密码 → 更新 password_hash
}
```

## 5 权限控制子模块（Casbin）

### 5.1 Casbin RBAC 模型

**文件：** `conf/rbac_model.conf`

```ini
[request_definition]
r = sub, obj, act

[policy_definition]
p = sub, obj, act

[role_definition]
g = _, _

[policy_effect]
e = some(where (p.eft == allow))

[matchers]
m = g(r.sub, p.sub) && keyMatch2(r.obj, p.obj) && regexMatch(r.act, p.act)
```

**匹配器说明：**
- `g(r.sub, p.sub)` — 检查用户角色是否匹配策略中的角色
- `keyMatch2(r.obj, p.obj)` — URL 路径通配符匹配（`/api/v1/users/:id` 匹配 `/api/v1/users*`）
- `regexMatch(r.act, p.act)` — HTTP 方法正则匹配

### 5.2 初始策略表

| ptype | v0 (sub) | v1 (obj) | v2 (act) | 说明 |
|-------|----------|----------|----------|------|
| p | super_admin | `/api/v1/*` | `.*` | 全权限 |
| p | lab_admin | `/api/v1/users*` | `.*` | 用户管理 |
| p | lab_admin | `/api/v1/equipments*` | `.*` | 设备管理 |
| p | lab_admin | `/api/v1/borrows*` | `.*` | 借阅管理 |
| p | lab_admin | `/api/v1/roles*` | `GET` | 角色只读 |
| p | lab_admin | `/api/v1/auth/logout` | `POST` | 登出 |
| p | lab_admin | `/api/v1/auth/refresh` | `POST` | 刷新 |
| p | member | `/api/v1/equipments*` | `GET` | 浏览设备 |
| p | member | `/api/v1/borrows/apply` | `POST` | 发起借阅 |
| p | member | `/api/v1/borrows/my` | `GET` | 我的借阅 |
| p | member | `/api/v1/borrows/\d+/return` | `POST` | 归还 |
| p | member | `/api/v1/borrows/\d+/cancel` | `POST` | 取消申请 |
| p | member | `/api/v1/users/\d+/password` | `PUT` | 改自己密码 |
| p | member | `/api/v1/roles` | `GET` | 角色列表 |
| p | member | `/api/v1/auth/logout` | `POST` | 登出 |
| p | member | `/api/v1/auth/refresh` | `POST` | 刷新 |

**角色分组策略说明：**

本系统 `Enforce()` 调用时传入的 `sub` 参数是 **角色名称**（`role_name`，如 `"member"`），而非用户名。Casbin matcher 中 `g(r.sub, p.sub)` 要求 `r.sub` 通过 `g` 函数映射到 `p.sub`。因此需要以下自映射策略使 RBAC 匹配器正常工作：

```csv
g, member, member           # 角色名 member 映射到策略主体 member
g, lab_admin, lab_admin     # 角色名 lab_admin 映射到策略主体 lab_admin
g, super_admin, super_admin # 角色名 super_admin 映射到策略主体 super_admin
```

> **为什么不用用户名作为 sub？** 当前设计中 JWT Claims 只携带 `role_name`，不携带角色层级关系。Casbin 中间件直接从 `c.GetString("role_name")` 获取 sub，无需额外查库。代价是 `g` 策略退化为恒等映射，无法表达"一个用户拥有多个角色"或"角色继承"。如果未来需要支持角色继承（如 super_admin 继承 lab_admin 的全部权限），需改为传递 `user_id` 作为 sub 并建立 `g, user_id, role` 映射。

### 5.3 Casbin 中间件

**位置：** `router/middleware/casbin.go`

**中间件链执行顺序（关键）：**
```
Recovery → Logger → Auth (JWT 解析 → 注入 role_name) → Casbin (读取 role_name → Enforce) → Controller
```

**处理流程：**
```
从 Context 获取 claims.role_name
  sub = role_name        // 如 "member"
  obj = Request.URL.Path  // 如 "/api/v1/borrows/1/approve"
  act = Request.Method    // 如 "POST"
  → enforcer.Enforce(sub, obj, act)
    ├── true  → c.Next()
    └── false → 记录 Warn 日志 + 403 {code:2005, msg:"权限不足"}
```

**越权告警日志格式：**
```json
{
    "level": "warn", "msg": "permission_denied",
    "user_id": 3, "username": "zhangsan", "role": "member",
    "path": "/api/v1/borrows/1/approve", "method": "POST", "ip": "192.168.1.100"
}
```

### 5.4 Casbin 初始化

```go
// cmd/main.go
adapter, _ := gormadapter.NewAdapterByDB(gormDB)
enforcer, _ := casbin.NewEnforcer("conf/rbac_model.conf", adapter)
enforcer.LoadPolicy()

// 种子数据初始化策略
func initCasbinPolicies(enforcer *casbin.Enforcer) {
    enforcer.AddPolicy("super_admin", "/api/v1/*", ".*")
    // ... 其他策略
    enforcer.AddGroupingPolicy("member", "member")
    enforcer.AddGroupingPolicy("lab_admin", "lab_admin")
    enforcer.AddGroupingPolicy("super_admin", "super_admin")
    enforcer.SavePolicy()
}
```

- 策略变更后调用 `enforcer.LoadPolicy()` 热加载，无需重启

## 8 安全考虑

1. **密码存储：** bcrypt cost=12，`password_hash` 标记 `json:"-"` 永不序列化
2. **Token 黑名单：** 登出/刷新时旧 Token 即时失效，Redis Key 自动过期
3. **密钥管理：** 生产环境通过 `LAB_JWT_SECRET` 环境变量注入，不写 config.yaml
4. **签名算法：** HMAC-SHA256
5. **过期策略：** 2 小时过期 + refresh 机制实现长期会话
6. **越权告警：** 所有 403 响应同时记录 Warn 日志，可对接告警系统

## 9 微服务演进说明

当前 IAM 模块在单体架构中承载三个子域（认证、用户管理、权限控制），因为它们共享 `sys_users` + `sys_roles` + `casbin_rule` 三张表。如果未来拆分微服务，合理的拆分方式是：

- **Auth Service：** 只持有 `username + password_hash + status`，负责登录/登出/JWT 签发
- **User Service：** 持有 `real_name + email + phone + role_id`，负责用户 CRUD
- **权限策略：** 随 Auth Service 或作为独立的 Policy Decision Point (PDP)

但在拆分前必须先拆表——当前 `sys_users` 一张表混合了认证信息和业务档案信息。
