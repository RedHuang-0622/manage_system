# 03 — 借阅工单管理

## 1 模块定位

借阅工单模块是整个系统的**核心业务模块**，承载"成员发起申请 → 负责人审批 → 系统扣减库存 → 归还恢复库存"的完整闭环。核心难点在于：**使用数据库事务保证库存扣减与记录生成的一致性，以及并发场景下防止设备超卖。**

> **设计约束：** 本模块与 [02-设备资产管理](02-设备资产管理.md) 的 `lab_equipments.available_stock` 字段存在 DB 级写耦合——审批和归还操作通过跨表事务同步修改设备库存。当前单体架构中借助 MySQL InnoDB 的行级锁 (`SELECT ... FOR UPDATE`) 保证并发安全，这是最优解。如果将来拆分为微服务，需要引入 Saga 模式替代跨服务 DB 事务。

**涉及的包/文件：**
- `controller/borrow.go` — HTTP 层
- `service/borrow.go` — 核心业务逻辑：事务编排、状态校验、并发控制
- `dao/borrow.go` — 数据访问层
- `models/borrow.go` — BorrowRecord 实体
- 跨模块依赖：
  - `dao/equipment.go` — `GetForUpdate` + `UpdateWithTx`（行锁 + 事务内写）
- **注入约定：** BorrowService 持有 `EquipmentDAO` 接口的引用（而非具体实现），便于单元测试时 mock 设备 DAO 的 FOR UPDATE 行为

## 2 数据模型

### 2.1 borrow_records 表 DDL

```sql
CREATE TABLE `borrow_records` (
    `id`           BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY,
    `user_id`      BIGINT UNSIGNED NOT NULL COMMENT '借用人ID',
    `equipment_id` BIGINT UNSIGNED NOT NULL COMMENT '设备ID',
    `quantity`     INT UNSIGNED    NOT NULL DEFAULT 1 COMMENT '借用数量',
    `status`       ENUM('申请中','已借出','已归还','被拒绝','已取消') NOT NULL DEFAULT '申请中',
    `apply_note`   VARCHAR(256)    NOT NULL DEFAULT '' COMMENT '申请备注',
    `approve_note` VARCHAR(256)    NOT NULL DEFAULT '' COMMENT '审批备注',
    `approver_id`  BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '审批人ID',
    `apply_at`     DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '申请时间',
    `approve_at`   DATETIME        DEFAULT NULL COMMENT '审批时间',
    `return_at`    DATETIME        DEFAULT NULL COMMENT '归还时间',
    `created_at`   DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at`   DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX `idx_user_id` (`user_id`),
    INDEX `idx_equipment_id` (`equipment_id`),
    INDEX `idx_status` (`status`),
    INDEX `idx_apply_at` (`apply_at`),
    CONSTRAINT `fk_borrow_user`  FOREIGN KEY (`user_id`)      REFERENCES `sys_users`(`id`),
    CONSTRAINT `fk_borrow_equip` FOREIGN KEY (`equipment_id`) REFERENCES `lab_equipments`(`id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='借阅工单表';
```

### 2.1.1 状态字段 ENUM vs TINYINT 选型

**选择 `ENUM` 的理由（V1.0）：**
- 本项目状态机仅 4 个状态，扩展频率低
- ENUM 值自带可读性，日志/调试时无需查映射表
- MySQL 8.0+ 支持 `ALTER TABLE ... MODIFY COLUMN ... ENUM(...)` 在末尾追加新值而不重建表

**已知风险和后续建议（V2.0）：**
- ENUM 修改中间值或删除值需要重建表
- 如果引入"逾期""续借中""部分归还"等中间状态，建议切换为 `TINYINT` + 应用层常量映射

| 方案 | DDL 写法 | 状态读取 |
|------|---------|---------|
| 当前 (ENUM) | `ENUM('申请中','已借出','已归还','被拒绝')` | 直接返回中文 |
| V2.0 建议 (TINYINT) | `TINYINT NOT NULL DEFAULT 0 COMMENT '0=申请中 1=已借出 2=已归还 3=被拒绝'` | 应用层映射为中文 |

### 2.2 索引设计说明

| 索引 | 用途 |
|------|------|
| `idx_user_id` | "我的借阅"查询 |
| `idx_equipment_id` | 设备关联借阅检查、下架前的未归还校验 |
| `idx_status` | 待审批列表（status='申请中'）、活跃借阅检查 |
| `idx_apply_at` | 按申请时间排序 |

## 3 状态机

```
                    ┌─────────────────────────────────────┐
                    │                                     │
                    ▼                                     │
              ┌──────────┐    审批通过     ┌──────────┐   归还    ┌──────────┐
  发起申请 ──> │  申请中   │ ────────────> │  已借出   │ ────────> │  已归还   │
              └────┬──┬──┘               └──────────┘            └──────────┘
                   │  │
         审批拒绝   │  │ 取消申请
                   │  │ (仅本人)
                   ▼  ▼
              ┌──────────┐   ┌──────────┐
              │  被拒绝   │   │  已取消   │  (终态)
              └──────────┘   └──────────┘
```

**状态转换表：**

| 当前状态 | 操作 | 目标状态 | 副作用 | 操作人 |
|---------|------|---------|--------|--------|
| 申请中 | 审批通过 | 已借出 | 扣减库存（跨表事务） | lab_admin+ |
| 申请中 | 审批拒绝 | 被拒绝 | 无 | lab_admin+ |
| 申请中 | 取消申请 | 已取消 | 无 | 仅本人 |
| 已借出 | 归还 | 已归还 | 恢复库存（跨表事务） | 本人/lab_admin+ |

**终态：** "被拒绝"、"已取消"和"已归还"不可再流转。

## 4 接口定义

### 4.1 发起借阅申请

```
POST /api/v1/borrows/apply
```

**权限：** 所有登录用户

**请求 Body：**
```json
{
    "equipment_id": 1,
    "quantity":     1,
    "apply_note":   "用于深度学习课程实验"
}
```

**Validator：**
- `equipment_id`: `required,gt=0`
- `quantity`: `required,gt=0`
- `apply_note`: `max=256`

**Service 层处理流程：**
1. 校验设备存在且 `status = 1`（上架）
2. 校验 `available_stock >= quantity`（初步检查，不做最终保障）
3. 检查用户是否已有同一设备的活跃借阅（status IN ('申请中','已借出')）
4. 创建 BorrowRecord，status = "申请中"

**成功响应 (201)：**
```json
{
    "code": 0, "msg": "申请已提交，等待审批",
    "data": {"id": 100, "equipment_id": 1, "quantity": 1, "status": "申请中", "apply_at": "..."}
}
```

**失败响应：**
```json
{"code": 3004, "msg": "设备库存不足", "data": null}
{"code": 3005, "msg": "您已有该设备的未归还借阅", "data": null}
```

### 4.2 审批借阅申请 ⭐ 核心事务

```
POST /api/v1/borrows/{id}/approve
```

**权限：** super_admin, lab_admin

**请求 Body：**
```json
{"approve": true, "approve_note": "同意借用，请按时归还"}
```

**核心实现：**

```go
func (s *BorrowService) Approve(ctx context.Context, borrowID uint,
    approve bool, approverID uint, note string) error {

    err := s.db.Transaction(func(tx *gorm.DB) error {
        // ① 锁定工单行
        record, err := s.borrowDAO.GetForUpdate(tx, borrowID)
        if err != nil { return ErrRecordNotFound }
        if record.Status != BorrowStatusApplied { return ErrInvalidStatus }

        // ② 锁定设备库存行（跨表行锁，防超卖）
        equip, err := s.equipDAO.GetForUpdate(tx, record.EquipmentID)
        if err != nil { return ErrEquipmentNotFound }

        // ③ 扣减库存 OR 拒绝
        now := time.Now()
        if approve {
            if equip.AvailableStock < record.Quantity { return ErrInsufficientStock }
            equip.AvailableStock -= record.Quantity
            record.Status = BorrowStatusBorrowed
            if err := s.equipDAO.UpdateWithTx(tx, equip); err != nil { return err }
        } else {
            record.Status = BorrowStatusRejected
        }

        // ④ 更新工单记录
        record.ApproverID = approverID
        record.ApproveNote = note
        record.ApproveAt = &now
        if err := s.borrowDAO.UpdateWithTx(tx, record); err != nil { return err }

        return nil // COMMIT（返回 error 则 ROLLBACK）
    })

    // ⑤ 事务 COMMIT 成功后再异步失效缓存（消除 COMMIT 前 DEL 的竞态窗口）
    if err == nil {
        // 此处 borrowID → equipmentID 的映射需从 record 传递出闭包
        // 实际实现中需在 Approve 签名中接收 equipmentID 或返回 record 供此使用
        go s.invalidateEquipmentCache(ctx, borrowID)
    }
    return err
}
```

**事务执行流程图：**

```
BEGIN TRANSACTION
  → SELECT * FROM borrow_records WHERE id=? FOR UPDATE      (锁工单)
     → 不存在 / status != "申请中" → ROLLBACK
  → SELECT * FROM lab_equipments WHERE id=? FOR UPDATE       (锁设备)
     → 不存在 → ROLLBACK
  → IF approve:
       available_stock < quantity → ROLLBACK (库存不足)
       available_stock -= quantity
       status = "已借出"
    ELSE:
       status = "被拒绝"
  → UPDATE lab_equipments SET available_stock=? WHERE id=?
  → UPDATE borrow_records SET status=?, approver_id=?, approve_at=? WHERE id=?
COMMIT (释放所有行锁)
  → 异步失效设备缓存
```

**成功响应 (200)：**
```json
{"code": 0, "msg": "审批通过，库存已扣减", "data": {"id": 100, "status": "已借出", "approve_at": "..."}}
```

**失败响应：**
```json
{"code": 3006, "msg": "库存不足，无法借出", "data": null}
{"code": 3007, "msg": "工单状态异常，无法审批", "data": null}
```

### 4.3 归还设备

```
POST /api/v1/borrows/{id}/return
```

**权限：** 借用人本人、lab_admin、super_admin

```go
func (s *BorrowService) Return(ctx context.Context, borrowID uint, operatorID uint, isAdmin bool) error {
    return s.db.Transaction(func(tx *gorm.DB) error {
        record, _ := s.borrowDAO.GetForUpdate(tx, borrowID)
        if record.Status != BorrowStatusBorrowed { return ErrInvalidStatus }

        // ⚠ 本人校验：非管理员只能归还自己的借阅（Casbin 无法区分 URL :id 是否为自己）
        if !isAdmin && record.UserID != operatorID {
            return ErrPermissionDenied
        }

        equip, _ := s.equipDAO.GetForUpdate(tx, record.EquipmentID)

        // 恢复库存
        equip.AvailableStock += record.Quantity
        s.equipDAO.UpdateWithTx(tx, equip)

        // 更新状态
        now := time.Now()
        record.Status = BorrowStatusReturned
        record.ReturnAt = &now
        s.borrowDAO.UpdateWithTx(tx, record)

        return nil // COMMIT
    })
    // 事务 COMMIT 成功后再失效缓存，消除缓存在 ROLLBACK 前被清除的窗口
    // 见 §9 第 6 条优化方案
}
```

**成功响应 (200)：**
```json
{"code": 0, "msg": "归还成功", "data": {"id": 100, "status": "已归还", "return_at": "..."}}
```

> **缓存失效时序说明：** 归还操作在事务 COMMIT 成功后异步失效设备缓存，不阻塞 HTTP 响应，详情参见 §9 第 6 条。

```go
func (s *BorrowService) Cancel(ctx context.Context, borrowID uint, operatorID uint) error {
    return s.db.Transaction(func(tx *gorm.DB) error {
        record, err := s.borrowDAO.GetForUpdate(tx, borrowID)
        if err != nil { return ErrRecordNotFound }
        if record.Status != BorrowStatusApplied { return ErrInvalidStatus }

    })
}
```

## 5 并发防超卖策略

### 5.1 方案选型

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|---------|
| **悲观锁 (SELECT FOR UPDATE)** ✅ | 强一致性，不会超卖 | 锁等待期间阻塞其他事务 | 并发冲突概率高 |
| 乐观锁 (version字段) | 无锁等待 | 冲突需重试，代码复杂 | 并发冲突概率低 |

**选择悲观锁的理由：** 热门设备的审批请求可能在短时间内集中到达（如实验课开始前），并发冲突概率高。

### 5.2 锁顺序与死锁预防

```
锁定顺序（所有事务统一遵守）：
① borrow_records 行   ← 先锁工单
② lab_equipments 行   ← 再锁设备
```

统一顺序 → 无循环等待 → 不死锁。

### 5.3 并发场景验证

```
假设设备 A: available_stock = 3

T1: 事务1 审批工单#100 (quantity=2)
    → 锁 borrow_records#100 → 锁 equipments#A
    → available=3 >= 2 ✓ → available=1 → COMMIT

T2: 事务2 审批工单#101 (quantity=1)
    → 锁 borrow_records#101 → 锁 equipments#A → 等待事务1...

T3: 事务1 COMMIT → 释放 equipments#A 锁

T4: 事务2 获取锁 → available=1 >= 1 ✓ → available=0 → COMMIT

T5: 事务3 审批工单#102 (quantity=2)
    → 等待事务2... → 获取锁 → available=0 < 2 ✗ → ROLLBACK
    → 返回 "库存不足" ✓ (成功防止超卖)
```

### 5.4 DAO 层 FOR UPDATE 查询

```go
// dao/borrow.go
func (d *BorrowDAO) GetForUpdate(tx *gorm.DB, id uint) (*models.BorrowRecord, error) {
    var record models.BorrowRecord
    err := tx.Clauses(clause.Locking{Strength: "UPDATE"}).
        Where("id = ?", id).First(&record).Error
    return &record, err
}

// dao/equipment.go
func (d *EquipmentDAO) GetForUpdate(tx *gorm.DB, id uint) (*models.LabEquipment, error) {
    var equip models.LabEquipment
    err := tx.Clauses(clause.Locking{Strength: "UPDATE"}).
        Where("id = ?", id).First(&equip).Error
    return &equip, err
}
```

## 8 错误码

| Code | 常量 | 含义 |
|------|------|------|
| 3004 | ErrStockInsufficient | 设备库存不足 |
| 3005 | ErrDuplicateBorrow | 已有该设备的未归还借阅 |
| 3006 | ErrBorrowApproveFailed | 审批失败（库存不足） |
| 3007 | ErrBorrowStatusInvalid | 工单状态异常 |

## 9 关键设计决策

1. **审批/归还使用跨表事务：** 利用 MySQL InnoDB 的 ACID 特性。GORM `db.Transaction` 中返回 error 自动回滚，返回 nil 自动提交
2. **申请时不扣库存：** 仅审批通过时扣减，避免"占坑"导致库存虚低
3. **允许申请人在审批前取消：** 本人可撤销"申请中"状态的工单，避免垃圾数据积累
4. **缓存失效在事务 COMMIT 后异步执行：** 事务成功返回后再 `go invalidateCache()`，消除事务回滚前 Redis 已被 DEL 的窗口
5. **禁止重复借阅：** 同一用户对同一设备同时只能有一笔活跃借阅
6. **锁定顺序固定：** borrow_records → lab_equipments，杜绝死锁
7. **归还/取消的本人校验在 Service 层：** Casbin 无法感知 URL 中 `:id` 是否属于当前用户，精确校验由 Service 层比对 `operatorID` 完成
