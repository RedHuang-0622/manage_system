# 00 — 系统概述与总览

## 1 项目背景与目标

本文档集合基于《设计文档摘要》和《PRD》，对"先进计算与知识工程实验室资产与人员管理系统"进行模块级设计。目标是为规范高校实验室的日常管理，解决人员权限混乱、设备借还记录不透明等问题，通过标准化 API 提供高并发、高安全的后端服务。

## 2 技术栈映射

| 职责 | 技术选型 | 说明 |
|------|---------|------|
| Web 框架 | Gin v1.9+ | 路由分发、请求上下文、中间件链 |
| ORM | GORM v2+ | 连接池化、多表关联、事务、行锁 |
| 数据库 | MySQL 8.0+ | InnoDB 引擎，支持行级锁与事务 |
| 缓存 | Redis 7+ | 设备列表缓存、JWT 黑名单 |
| 配置管理 | Viper | 解析 config.yaml，环境变量覆盖 |
| 身份认证 | JWT (golang-jwt) | Token 签发/解析，配置过期时间 |
| 权限控制 | Casbin v2 + GORM Adapter | RBAC 模型，策略持久化到 MySQL |
| 参数校验 | Validator (go-playground/validator) | 结构体 Tag 校验 |
| 日志 | Zap (uber-go/zap) | 结构化 JSON 日志，按天切割 |
| 密码加密 | bcrypt | cost=12 |

> **跨模块耦合共三处：**
> 1. **01 → 02/03：** Auth/Casbin 中间件拦截所有受保护路由
> 2. **02 ↔ 03：** DB 级事务耦合（`lab_equipments.available_stock`）
> 3. **03 → 01：** FK 依赖——`borrow_records.user_id` 和 `borrow_records.approver_id` 引用 `sys_users.id`，Preload 时需 JOIN
>
> 02 ↔ 03 的 DB 事务耦合是当前单体架构的合理权衡。详见两份文档中的"设计约束"说明。

## 7 依赖注入与启动流程

```go
// cmd/main.go
func main() {
    cfg    := config.Load("conf/config.yaml")              // 1. 配置
    logger := zap.InitLogger(cfg.Log)                      // 2. 日志
    db     := initDB(cfg.MySQL)                            // 3. 数据库
    db.AutoMigrate(&models.SysUser{}, &models.SysRole{},
        &models.LabEquipment{}, &models.BorrowRecord{})
    seedData(db)
    rdb      := redis.InitRedis(cfg.Redis)                 // 4. Redis
    enforcer := initCasbin(db, cfg.Casbin.ModelPath)       // 5. Casbin
    deps     := Dependencies{DB: db, Redis: rdb,           // 6. 组装
        Logger: logger, Enforcer: enforcer, Config: cfg}
    initLayers(deps)                                       // 7. DAO→Service→Controller
    r := router.SetupRouter(deps)                          // 8. 路由
    srv := &http.Server{Addr: fmt.Sprintf(":%d", cfg.Server.Port), Handler: r}

    // 9. 启动 + Graceful Shutdown
    go func() {
        logger.Info("server_started", zap.Int("port", cfg.Server.Port))
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            logger.Fatal("server_failed", zap.Error(err))
        }
    }()
    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    <-quit
    logger.Info("server_shutting_down")
    ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
    defer cancel()
    srv.Shutdown(ctx)           // 等待进行中的请求完成
    sqlDB, _ := db.DB(); sqlDB.Close()
    rdb.Close()
    logger.Info("server_stopped")
}
```

## 10 测试计划概要

| 层级 | 测试内容 | 工具 |
|------|---------|------|
| DAO | CRUD、分页、FOR UPDATE 锁 | go-sqlmock |
| Service | 业务逻辑、事务提交/回滚、边界条件 | go-sqlmock + testify |
| pkg/jwt | Token 签发、解析、过期、黑名单 | testify |
| Controller | 参数校验、响应格式 | httptest + testify |

**集成测试场景：**
1. 完整借阅闭环（登录→创建用户→创建设备→申请→审批→验证库存→归还→验证恢复）
2. 并发审批同一设备（库存 3，10 并发，验证仅 3 通过）
3. 权限越界（member 调审批接口 → 403）
4. Token 黑名单（登出后旧 Token → 401）
5. 缓存一致性（写库后验证缓存失效）
