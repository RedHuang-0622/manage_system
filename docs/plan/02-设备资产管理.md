# 02 — 设备资产管理

## 1 模块定位

设备资产模块负责实验室设备的全生命周期管理：入库、信息更新、下架，以及**设备大厅**的高频列表查询。为应对高并发读场景，引入 Redis 缓存优化查询性能，写操作后主动失效缓存。

> **设计约束（与借阅模块的耦合）：** 本模块的 `available_stock` 字段有两类写入路径——入库/更新由本模块直接写入；审批/归还由借阅模块通过同一个 MySQL DB 事务写入（利用 `SELECT ... FOR UPDATE` 保证一致性）。在当前单体架构中这是最优解，如果将来拆分为微服务，需要引入 Saga/Outbox 模式替代跨服务 DB 事务。

**涉及的包/文件：**
- `controller/equipment.go` — HTTP 层
- `service/equipment.go` — 业务逻辑：库存管理、缓存编排
- `dao/equipment.go` — 数据访问层：查询、行锁
- `models/equipment.go` — LabEquipment 实体

## 2 数据模型

### 2.1 lab_equipments 表 DDL

```sql
CREATE TABLE `lab_equipments` (
    `id`              BIGINT UNSIGNED NOT NULL AUTO_INCREMENT PRIMARY KEY,
    `name`            VARCHAR(128)    NOT NULL COMMENT '设备名称',
    `model`           VARCHAR(64)     NOT NULL DEFAULT '' COMMENT '设备型号',
    `category`        VARCHAR(32)     NOT NULL DEFAULT '' COMMENT '分类',
    `total_stock`     INT UNSIGNED    NOT NULL DEFAULT 0 COMMENT '总库存',
    `available_stock` INT UNSIGNED    NOT NULL DEFAULT 0 COMMENT '可用库存',
    `location`        VARCHAR(64)     NOT NULL DEFAULT '' COMMENT '存放位置',
    `description`     TEXT            COMMENT '设备描述/备注',
    `status`          TINYINT         NOT NULL DEFAULT 1 COMMENT '1=上架 0=下架',
    `created_at`      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP,
    `updated_at`      DATETIME        NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    INDEX `idx_name` (`name`),
    INDEX `idx_category` (`category`),
    INDEX `idx_status_available` (`status`, `available_stock`),
    FULLTEXT INDEX `ft_name_description` (`name`, `description`) WITH PARSER ngram
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='实验室设备表';
```

| 索引 | 用途 |

| `idx_category` | 按分类筛选 |

**库存联动规则：**

> ⚠️ `total_stock` 和 `available_stock` 定义为 `uint`，直接相减在下调 total_stock 时会发生无符号整数下溢（如 `5 - 10` 变成极大值）。必须先比大小再转为有符号计算。

```go
// total_stock、available_stock 均为 uint，须避免无符号下溢
delta := int(newTotal) - int(oldTotal)            // 可能为负
newAvailable := int(oldAvailable) + delta
if newAvailable < 0 { return ErrStockInsufficient }
equip.TotalStock = newTotal
equip.AvailableStock = uint(newAvailable)
```

**副作用：** 失效详情缓存 + 批量失效列表缓存

### 3.5 下架设备

```
DELETE /api/v1/equipments/{id}
```

**权限：** super_admin

**实现：** 软删除 → `status = 0`

**关联检查：** 检查是否有未归还借阅，必须与 status 更新在同一事务中加行锁，防止 TOCTOU 竞态：

```go
func (s *EquipmentService) Disable(ctx context.Context, id uint) error {
    return s.db.Transaction(func(tx *gorm.DB) error {
        // ① 锁定设备行（FOR UPDATE）
        equip, err := s.dao.GetForUpdate(tx, id)
        if err != nil { return ErrEquipmentNotFound }
        if equip.Status == 0 { return nil } // 幂等：已下架

        // ② 在同一事务内检查未归还借阅（消除 TOCTOU 窗口）
        count, err := s.borrowDAO.CountByEquipmentAndStatusInTx(tx, id, "已借出")
        if err != nil { return err }
        if count > 0 { return errors.New("设备有未归还借阅，无法下架") }

        // ③ 更新状态
        equip.Status = 0
        if err := s.dao.UpdateWithTx(tx, equip); err != nil { return err }

        return nil // COMMIT → 异步失效缓存
    })
}
```

**副作用：** 失效详情缓存 + 批量失效列表缓存（在事务 COMMIT 成功后启动 goroutine）

## 4 Redis 缓存策略

### 4.1 缓存流程图

```
读请求（列表/详情）
  → 查 Redis → 命中 → 返回
           → 未命中 → 查 MySQL → 写 Redis → 返回

写请求（创建/更新/下架）
  → 写 MySQL → 异步 goroutine 失效相关 Redis Key → 返回
```

### 4.2 缓存 Key 规范

| 场景 | Key 格式 | TTL | 说明 |
|------|---------|-----|------|
| 设备列表 | `equip:list:{page}:{size}:{kw_hash}:{cat}:{status}:{avail}` | 30s | 短 TTL 保证新鲜度 |
| 设备详情 | `equip:detail:{id}` | 60s | 详情变更频率低 |

**Key 示例：**
```
equip:list:1:12:abc123::1:0    → 第1页,12条,kw_hash=abc123,无分类,上架,不限可用
equip:detail:42                 → 设备ID=42
```

### 4.3 缓存读取实现

```go
func (s *EquipmentService) ListPage(ctx context.Context, req *ListEquipReq) (*PageResult, error) {
    cacheKey := buildListCacheKey(req)

    // 1. 读缓存
    cached, err := s.redis.Get(ctx, cacheKey).Result()
    if err == nil {
        var result PageResult
        json.Unmarshal([]byte(cached), &result)
        return &result, nil
    }

    // 2. 查 DB
    equipments, total, err := s.dao.FindPage(...)
    result := &PageResult{Total: total, List: equipments, Page: req.Page, PageSize: req.PageSize}

    // 3. 写缓存
    data, _ := json.Marshal(result)
    s.redis.Set(ctx, cacheKey, data, 30*time.Second)
    return result, nil
}
```

### 4.4 缓存失效策略

```go
func (s *EquipmentService) invalidateDetailCache(id uint) {
    s.redis.Del(ctx, fmt.Sprintf("equip:detail:%d", id))
}
func (s *EquipmentService) invalidateListCache() {
    // SCAN 代替 KEYS，避免阻塞 Redis
    iter := s.redis.Scan(ctx, 0, "equip:list:*", 100).Iterator()
    for iter.Next(ctx) { s.redis.Del(ctx, iter.Val()) }
}
```

### 4.5 缓存一致性保证

| 场景 | 策略 | 影响范围 |
|------|------|---------|
| 创建/更新/下架设备 | 写 DB 后异步失效 | `equip:detail:{id}` + `equip:list:*` |
| 借阅审批扣库存 | 事务回调内 goroutine 异步失效（与 COMMIT 存在竞态窗口，窗口 ≤ 30s TTL） | 同上 |
| TTL 自然过期 | 30s/60s 后自动重建 | 无 |

### 4.6 缓存一致性的已知局限

跨模块写操作（借阅模块修改设备库存）的缓存失效存在竞态窗口：

```
时间线：
  T0: 事务内 UPDATE lab_equipments.available_stock
  T1: goroutine.DEL(equip:detail:{id})    ← 可能发生在 COMMIT 前
  T2: GORM COMMIT
```

如果 T1 在 T2 之前执行，且 T1-T2 之间有另一个请求读到旧库存并回填缓存，则缓存中保存的是旧值（直到 TTL 过期或下次写入触发失效）。

**缓解措施：**
- 列表缓存 TTL 仅 30s，最坏情况下 30s 后自动修复
- 生产环境建议将缓存失效移到事务 COMMIT 成功之后（见下方优化方案）

**优化方案（V1.1）：**
```go
err := s.db.Transaction(func(tx *gorm.DB) error { /* ... */ return nil })
if err == nil {
    // 事务已 COMMIT，此时失效缓存无竞态窗口
    go s.invalidateEquipmentCache(ctx, equipID)
}
return err
```

## 5 库存写入路径说明

`available_stock` 字段有两个写入来源：

```
┌──────────────────────────┐     ┌──────────────────────────┐
│    本模块 (Equipment)     │     │  借阅模块 (Borrow)        │
│                          │     │                          │
│  创建设备 → avail=total   │     │  审批通过 → avail -= qty │
│  更新设备 → avail+=delta  │     │  归还    → avail += qty │
│                          │     │                          │
│  (独立 DB 操作)           │     │  (同一 DB 事务中)        │
└──────────────────────────┘     └──────────────────────────┘
```

**关键约束：** 借阅模块通过 `SELECT ... FOR UPDATE` 行锁 + GORM Transaction 保证并发写入安全，详见 [03-借阅工单管理](03-借阅工单管理.md) 第 5 章。

```go
type EquipmentDAO interface {

```

**行级锁实现：**
```go
func (d *EquipmentDAO) GetForUpdate(tx *gorm.DB, id uint) (*models.LabEquipment, error) {
    var equip models.LabEquipment
    err := tx.Clauses(clause.Locking{Strength: "UPDATE"}).
        Where("id = ?", id).First(&equip).Error
    return &equip, err
}
```

## 7.1 跨模块依赖契约

本模块有两处需要查询借阅模块的数据，通过以下接口解耦：

### 7.1.1 下架设备时检查未归还借阅

```
调用方: EquipmentService.Disable()
被调方: BorrowDAO
方法:   CountByEquipmentAndStatusInTx(tx *gorm.DB, equipID uint, status string) (int64, error)
        CountByEquipmentAndStatus(equipID uint, status string) (int64, error)
```

**契约：**
- 输入：设备 ID + 借阅状态（固定为 `"已借出"`）
- 输出：该设备"已借出"状态的记录数
- **关键约束：** `Disable()` 必须在同一事务中调用 `CountByEquipmentAndStatusInTx(tx, ...)`，与 `GetForUpdate` 共享事务上下文，消除 TOCTOU 窗口
- 非事务场景（如仅查询统计）使用 `CountByEquipmentAndStatus()` 即可

**依赖注入：**
```go
type EquipmentService struct {
    dao       EquipmentDAO
    db        *gorm.DB                   // ← 需要持有 DB 以开启事务
    borrowDAO BorrowStatusChecker        // ← 接口而非具体实现
    redis     *redis.Client
}

type BorrowStatusChecker interface {
    CountByEquipmentAndStatusInTx(tx *gorm.DB, equipID uint, status string) (int64, error)
    CountByEquipmentAndStatus(equipID uint, status string) (int64, error)
}
```

### 7.1.2 BorrowService 事务内写设备库存

```
调用方: BorrowService.Approve() / BorrowService.Return()
被调方: EquipmentDAO
方法:   GetForUpdate(tx *gorm.DB, id uint) (*LabEquipment, error)
        UpdateWithTx(tx *gorm.DB, equip *LabEquipment) error
```

**契约：**
- 必须在同一个 `*gorm.DB` 事务内调用
- 调用方负责锁定顺序：先锁 borrow_records，再锁 lab_equipments
- `GetForUpdate` 返回的实体由调用方修改 `AvailableStock` 后传回 `UpdateWithTx`

> **设计原则：** 借阅模块可以调用设备 DAO 的事务方法，但不直接操作设备 Service。库存扣减逻辑在 BorrowService 中编排，EquipmentDAO 只提供原子读写能力。

## 9 性能考虑

1. **列表缓存 TTL 30s：** 设备大厅高频查询，30 秒缓存显著降低 DB 压力
2. **详情缓存 TTL 60s：** 详情访问频率低，稍长 TTL 提升命中率
3. **写操作异步失效缓存：** goroutine 异步删除，不阻塞用户响应
4. **SCAN 代替 KEYS：** 避免阻塞 Redis 单线程
5. **联合索引 `idx_status_available`：** 覆盖"上架+有库存"查询，避免回表
